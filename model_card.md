# Model Card — PawPal+ AI Health Advisor

## Project Overview

**Project name:** PawPal+ with RAG-powered Health Q&A
**Base model used:** Claude via the Anthropic API
**AI feature type:** Retrieval-Augmented Generation (RAG)
**Interface:** Streamlit web app

---

## Intended Use

This model is used to answer natural language health and care questions about a specific pet, grounded in that pet's logged data. It is intended for use by pet owners who want personalized, context-aware guidance on their pet's daily care, behavior, or health symptoms.

**It is not intended to replace veterinary advice.** Every response generated by this system ends with a reminder to consult a licensed veterinarian for medical concerns.

---

## Limitations and Biases

**Breed coverage bias:** The knowledge base only covers popular Western breeds. Owners of less common breeds (e.g. Basenji, Sphynx, Abyssinian) receive the generic species-level response instead of breed-specific advice, which results in less personalized answers for those users.

**English only:** The system prompt and knowledge base are written in English. Non-English inputs may produce inconsistent or lower-quality results.

**No memory between sessions:** The system has no persistent memory. Each question is answered independently — it cannot reference anything said or logged in a previous session.

**Medical knowledge is general, not clinical:** The breed knowledge base was written as general care guidance, not veterinary reference material. It does not reflect the latest clinical research and should not be treated as medical fact.

**Socioeconomic bias in recommendations:** The knowledge base may reflect care standards that assume access to regular vet visits, premium food, and enrichment products — advice that is not equally accessible to all pet owners.

---

## Could This AI Be Misused? How Would You Prevent It?

Yes, there are realistic misuse scenarios:

**Replacing veterinary care:** A user could treat the AI's response as a diagnosis and delay or avoid taking a sick pet to a vet. This is the most likely and most serious misuse.
**Prevention:** Every single response ends with a hard-coded disclaimer: *"This is not a substitute for professional veterinary advice. Please consult a licensed veterinarian for any medical concerns."* The system prompt also explicitly instructs Claude never to diagnose conditions or prescribe treatments.

## What Surprised Me While Testing Reliability

The most surprising finding was how dramatically the specificity of the retrieved context affected response quality. When I tested with a complete pet profile (breed, age, medical condition, recent tasks), the responses were noticeably more useful and personalized. When I tested with a minimal profile (species only, no tasks), the responses were barely better than a generic search result — even though the same model was being used.

This was surprising because I assumed the model's general knowledge would compensate for thin context. It didn't. The retrieval step turned out to be more important than expected.

A second surprise was inconsistency across repeated identical questions. Asking the same question about the same pet twice produced responses that agreed on key points but used different phrasing and occasionally different orderings of advice. This is expected behavior for a probabilistic model, but seeing it in practice made it clear why a reliability testing system — even a simple one that checks whether the pet's name and medical condition appear in every response — would be a meaningful addition.

A third surprise was how well the model handled edge cases I hadn't explicitly designed for. When I asked a vague question like "is my pet ok?", Claude still referenced the pet's profile and gave a reasonable answer rather than hallucinating or crashing. The system prompt structure was doing more work than I realized.

## AI Collaboration During This Project

AI tools were used throughout this project in a genuine collaborative role — not just for code generation, but for thinking through design decisions, structuring documentation, and debugging problems.

**One instance where the AI gave a genuinely helpful suggestion:**
When brainstorming how to implement RAG, Claude suggested using the existing `Pet` dataclass fields directly as the retrieval source — rather than setting up a separate vector database or embedding store. This was the right call for the project's scope. It kept the implementation simple, kept the context grounded in live data, and avoided adding infrastructure that wasn't needed. I hadn't thought of framing the existing data model as the "retriever" — that reframing made the whole feature easier to build and easier to explain to a grader.


this experience reinforced that AI-generated code needs to be reviewed by someone who understands the broader system constraints, not just whether the code runs without errors.

## Testing Results

**What worked:**
- RAG context reliably influenced responses — medical conditions in the pet profile appeared in AI responses without being explicitly mentioned in the question
- Fallback logic worked correctly for unknown breeds — the system returned species-level advice instead of erroring
- The veterinary disclaimer appeared in every response tested
- Error handling worked — a missing API key and depleted credits both showed user-friendly messages instead of crashes

**What didn't work as expected:**
- Vague questions produced generic responses even with a specific pet profile in context
- Minor response inconsistency across repeated identical questions
- No automated test exists for AI output content — verification was manual only

**Edge cases not yet tested:**
- A pet with no tasks logged (empty context)
- Questions in languages other than English
- Extremely long medical condition lists approaching token limits

> *What this project says about me as an AI engineer:*

This project showed me that building with AI is fundamentally a systems design problem, not just a prompting problem. The part that took the most thought wasn't writing the Claude API call. it was deciding what to retrieve, how to structure the context, where to put the human review step, and how to make sure the AI couldn't be misused in ways that would harm someone's pet. i care about building things that are useful in practice, not just technically functional in a demo. The fact that I added a veterinary disclaimer, error handling, and a fallback for unknown breeds — even though none of those were required — reflects how I think about what it means to ship something responsibly. I want to keep building AI systems that are honest about what they can and can't do.
